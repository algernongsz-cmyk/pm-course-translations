Coding Agent 的组成部分

Coding agent 如何使用工具、内存和仓库上下文让 LLM 在实践中更好地工作

作者：Sebastian Raschka, PhD | 翻译：AI | 原文：Components of A Coding Agent

在这篇文章中，我想介绍 coding agent 和 agent harness 的整体设计：它们是什么、如何工作，以及各个部分在实践中如何组合在一起。我的《Build a Large Language Model (From Scratch)》和《Build a Large Reasoning Model (From Scratch)》两本书的读者经常问起关于 agent 的问题，所以我认为写一篇可以引用的参考文章会很有用。

更广泛地说，agent 已成为一个重要话题，因为近期实用 LLM 系统的许多进步不仅来自更好的模型，还来自我们如何使用它们。在许多实际应用中，周围系统（如工具使用、上下文管理和内存）扮演着与模型本身同样重要的角色。这也解释了为什么 Claude Code 或 Codex 等系统感觉比在普通聊天界面中使用的相同模型更强大。

在本文中，我列出了 coding agent 的六个主要构建模块。

Claude Code、Codex CLI 和其他 Coding Agent

你可能已经熟悉 Claude Code 或 Codex CLI，但为了铺垫背景，它们本质上是将 LLM 包装在应用层中的 agentic 编码工具——所谓的 agentic harness——以便为编码任务提供更方便、更高性能的体验。

Coding agent 是专为软件工作设计的，其显著特点不仅在于模型选择，还在于周围系统，包括仓库上下文、工具设计、提示缓存稳定性、内存和长会话连续性。

这种区别很重要，因为当我们谈论 LLM 的编码能力时，人们经常将模型、推理行为和 agent 产品混为一谈。但在深入 coding agent 的细节之前，让我先简要介绍一下 LLM、推理模型和 agent 这些更广泛概念之间的区别。

LLM、推理模型和 Agent 之间的关系

LLM 是核心的 next-token 模型。推理模型仍然是 LLM，但通常是经过训练和/或提示，在中间推理、验证或候选答案搜索上花费更多推理时计算的模型。

Agent 是上面的一层，可以理解为围绕模型的控制循环。通常，给定一个目标，agent 层（或 harness）决定下一步检查什么、调用哪些工具、如何更新其状态以及何时停止等。

粗略地说，我们可以这样理解这种关系：LLM 是引擎，推理模型是增强版引擎（更强大，但使用成本更高），而 agent harness 帮助我们使用模型。这个比喻并不完美，因为我们也可以将传统 LLM 和推理 LLM 作为独立模型使用（在聊天 UI 或 Python 会话中），但我希望它能传达主要观点。

换句话说，agent 是在环境中重复调用模型的系统。

简而言之，我们可以这样总结：

  • LLM：原始模型
  • 推理模型：优化后输出中间推理轨迹并进行更多自我验证的 LLM
  • Agent：使用模型加上工具、内存和环境反馈的循环
  • Agent harness：围绕 agent 的软件支架，管理上下文、工具使用、提示、状态和控制流
  • Coding harness：agent harness 的特例；即用于软件工程的特定任务 harness，管理代码上下文、工具、执行和迭代反馈

如上所述，在 agent 和编码工具的语境中，我们还有两个流行术语 agent harness 和（agentic）coding harness。Coding harness 是围绕模型的软件支架，帮助它有效地编写和编辑代码。Agent harness 则更广泛，不特定于编码（例如，想想 OpenClaw）。Codex 和 Claude Code 可以被视为 coding harness。

无论如何，更好的 LLM 为推理模型提供更好的基础（这涉及额外的训练），而 harness 从这个推理模型中获得更多。

当然，LLM 和推理模型本身也能够解决编码任务（没有 harness），但编码工作只是部分关于 next-token 生成。很多工作涉及仓库导航、搜索、函数查找、diff 应用、测试执行、错误检查以及将所有相关信息保持在上下文中。（程序员可能知道这是艰苦的脑力工作，这就是为什么我们不喜欢在编码会话中被打断 :)）。

这里的要点是，好的 coding harness 可以让推理模型和非推理模型感觉比在普通聊天框中强大得多，因为它有助于上下文管理等。

The Coding Harness

如前一节所述，当我们说 harness 时，我们通常指围绕模型的软件层，它组装提示、暴露工具、跟踪文件状态、应用编辑、运行命令、管理权限、缓存稳定前缀、存储内存等。

如今，使用 LLM 时，与直接提示模型或使用 Web 聊天 UI（更接近"与上传文件聊天"）相比，这一层塑造了大部分用户体验。

在我看来，当今 LLM 的原始版本具有非常相似的能力（例如，GPT-5.4、Opus 4.6 和 GLM-5 等的原始版本），harness 往往是使一个 LLM 比另一个更好的区分因素。

这是推测性的，但我怀疑如果我们将最新、最有能力的开源权重 LLM（如 GLM-5）放入类似的 harness 中，它可能会与 Codex 中的 GPT-5.4 或 Claude Code 中的 Claude Opus 4.6 表现相当。也就是说，一些特定于 harness 的后训练通常是有益的。例如，OpenAI 历史上维护了单独的 GPT-5.3 和 GPT-5.3-Codex 变体。

在下一节中，我想更深入地讨论使用我的 Mini Coding Agent：https://github.com/rasbt/mini-coding-agent 来介绍 coding harness 的核心组件。

顺便说一句，在本文中，为了简单起见，我某种程度上互换地使用术语"coding agent"和"coding harness"。（严格来说，agent 是模型驱动的决策循环，而 harness 是提供上下文、工具和执行支持的周围软件支架。）

无论如何，以下是 coding agent 的六个主要组件。你可以查看我的最小但完全可用的、从零开始的 Mini Coding Agent（用纯 Python 实现）的源代码，了解更多具体的代码示例。代码通过代码注释标注了下面讨论的六个组件：

1. Live Repo Context

这可能是最明显的组件，但也是最重要的组件之一。

当用户说"修复测试"或"实现 xyz"时，模型应该知道它是否在 Git 仓库中、在哪个分支上、哪些项目文档可能包含说明等。

这是因为这些细节通常会改变或影响正确的操作。例如，"修复测试"不是一个自包含的指令。如果 agent 看到 AGENTS.md 或项目 README，它可能会了解要运行哪个测试命令等。如果它知道仓库根目录和布局，它可以在正确的位置查找而不是猜测。

此外，git 分支、状态和提交可以帮助提供更多关于当前正在进行哪些更改以及在哪里关注的上下文。

2. Prompt Shape And Cache Reuse

一旦 agent 有了仓库视图，下一个问题是如何将该信息提供给模型。前面的图展示了一个简化的视图（"Combined prompt: prefix + request"），但在实践中，每次用户查询都合并和重新处理工作区摘要将相对浪费。

即，编码会话是重复的，agent 规则通常保持不变。工具描述通常也保持不变。甚至工作区摘要通常也（大部分）保持不变。主要变化通常是最新的用户请求、最近的转录和可能的短期记忆。

"智能"运行时不会在每一轮都重建所有内容作为一个巨大的未分化提示，如下图所示。

"稳定提示前缀"意味着其中包含的信息变化不大。它通常包含一般说明、工具描述和工作区摘要。如果没有重要变化，我们不想在每次交互中从头重建它而浪费计算。

其他组件更新更频繁（通常每轮）。这包括短期记忆、最近的转录和最新的用户请求。

简而言之，"稳定提示前缀"的缓存方面只是智能运行时尝试重用该部分。

3. Tool Access and Use

工具访问和工具使用是开始感觉不太像聊天而更像 agent 的地方。

普通模型可以用散文建议命令，但 coding harness 中的 LLM 应该做更窄、更有用的事情，并且实际上能够执行命令并检索结果（而不是我们手动调用命令并将结果粘贴回聊天）。

但是，harness 通常提供预定义的允许和命名工具列表，具有明确的输入和边界，而不是让模型即兴创作任意语法。（当然，Python subprocess.call

在这里，模型必须选择一个 harness 识别的操作，如列出文件、读取文件、搜索、运行 shell 命令、写入文件等。它还必须提供 harness 可以检查的形状的参数。

所以当模型要求做某事时，运行时可以停止并运行程序化检查，如：

  • "这是一个已知的工具吗？"
  • "参数有效吗？"
  • "这需要用户批准吗？"
  • "请求的路径甚至在工作区内吗？"

只有在这些检查通过后才实际运行任何东西。

虽然运行 coding agent 当然带有一些风险，但 harness 检查也提高了可靠性，因为模型不会执行完全任意的命令。

此外，除了拒绝格式错误的操作和批准门控外，文件访问可以通过检查文件路径保持在仓库内。

从某种意义上说，harness 给模型更少的自由，但它同时也提高了可用性。

4. Minimizing Context Bloat

上下文膨胀不是 coding agent 独有的问题，而是 LLM 普遍存在的问题。当然，如今 LLM 支持越来越长的上下文（我最近写了一篇关于使这在计算上更可行的 注意力变体），但长上下文仍然昂贵，并且可能引入额外的噪音（如果有大量不相关的信息）。

Coding agent 比多轮聊天期间的普通 LLM 更容易受到上下文膨胀的影响，因为重复的文件读取、冗长的工具输出、日志等。

如果运行时以全保真度保留所有这些，它将很快用完可用的上下文 token。所以，好的 coding harness 通常非常复杂地处理上下文膨胀，而不仅仅是像普通聊天 UI 那样切割或总结信息。

从概念上讲，coding agent 中的上下文压缩可能如下图所示。具体来说，我们进一步放大前一节图8中的 clip（步骤6）部分。

最小 harness 至少使用两种压缩策略来管理该问题。

第一种是裁剪，它缩短长文档片段、大型工具输出、记忆笔记和转录条目。换句话说，它防止任何一段文本仅仅因为它恰好冗长而接管提示预算。

第二种策略是转录减少或摘要，它将完整会话历史（下一节将详细介绍）转换为较小的可提示摘要。

这里的一个关键技巧是让最近的事件更丰富，因为它们更可能与当前步骤相关。我们更积极地压缩较早的事件，因为它们可能不太相关。

此外，我们还去重较早的文件读取，这样模型就不会仅仅因为在会话早期多次读取而不断看到相同的文件内容。

总的来说，我认为这是好的 coding agent 设计中被低估、乏味的部分之一。许多明显的"模型质量"实际上是上下文质量。

5. Structured Session Memory

在实践中，这里涵盖的所有这6个核心概念都是高度交织的，不同的部分和图以不同的焦点或缩放级别涵盖它们。在上一节中，我们涵盖了历史的提示时使用以及我们如何构建紧凑的转录。那里的问题是：在下一轮中应该有多少过去的信息返回给模型？所以重点是压缩、裁剪、去重和时效性。

现在这一节，结构化会话记忆，是关于历史的存储时结构。这里的问题是：agent 随着时间的推移保持什么作为永久记录？所以重点是运行时保持更完整的转录作为持久状态，以及一个更轻的记忆层，它更小，并且被修改和压缩而不是仅仅追加。

总之，coding agent 将状态分离为（至少）两层：

  • 工作记忆：agent 明确保持的小型、精炼状态
  • 完整转录：涵盖所有用户请求、工具输出和 LLM 响应

但紧凑转录和工作记忆有略微不同的工作。紧凑转录用于提示重建。它的工作是给模型一个压缩的最近历史视图，以便它可以继续对话而不必每轮看到完整转录。工作记忆更多用于任务连续性。它的工作是保持一个小型、明确维护的跨轮次重要事项摘要，如当前任务、重要文件和最近的笔记。

按照上图的步骤4，最新的用户请求，连同 LLM 响应和工具输出，将被记录为"新事件"在完整转录和工作记忆中，在下一轮，这未显示以减少上图中的混乱。

6. Delegation With (Bounded) Subagents

一旦 agent 有了工具和状态，下一个有用的能力之一是委托。

原因是它允许我们通过子代理将某些工作并行化为子任务并加速主任务。例如，主 agent 可能正在执行一个任务，但仍需要一个侧面答案，例如，哪个文件定义了一个符号、配置说什么或测试为什么失败。将其分离为一个有界的子任务很有用，而不是强迫一个循环一次携带每个工作线程。

（在我的 mini coding agent 中，实现更简单，子代仍然同步运行，但基本思想是相同的。）

子代理只有在继承了足够的上下文才能做真正的工作时才有用。但如果我们不限制它，我们现在有多个 agent 重复工作、触及相同的文件或产生更多子代理等。

所以棘手的设计问题不仅是如何产生子代理，还有如何约束它 :)

Claude Code 很早就支持子代理，Codex 最近也添加了它们。Codex 通常不强制子代理进入只读模式。相反，它们通常继承主 agent 的大部分沙箱和批准设置。所以，边界更多是关于任务范围、上下文和深度。

组件总结

上面的部分试图涵盖 coding agent 的主要组件。如前所述，它们在实现中或多或少深度交织。然而，我希望逐一涵盖它们有助于建立 coding harness 如何工作的整体心智模型，以及为什么与简单的多轮聊天相比，它们可以使 LLM 更有用。

这与 OpenClaw 相比如何？

OpenClaw 可能是一个有趣的比较，但它不是完全相同类型的系统。

OpenClaw 更像是一个本地、通用 agent 平台，也可以编码，而不是专门的（终端）编码助手。

与 coding harness 仍有一些重叠：

  • 它使用工作区中的提示和指令文件，如 AGENTS.md、SOUL.md 和 TOOLS.md
  • 它保持 JSONL 会话文件并包括转录压缩和会话管理
  • 它可以产生助手会话和子代理
  • 等等

然而，如上所述，重点是不同的。Coding agent 针对在仓库中工作并要求编码助手检查文件、编辑代码和高效运行本地工具的人进行了优化。OpenClaw 更针对在聊天、频道和工作区中运行许多长期存在的本地 agent 进行了优化，编码是其中几个重要工作负载之一。
