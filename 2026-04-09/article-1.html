几乎每个人在职业生涯的某个阶段都曾经历过那种极度沮丧的过程——将大量数据从一个地方移动到另一个地方。如果你还没有，那可能只是因为你还没有处理过足够大的数据集。对于 Andy Warfield 来说，他的一次重要经历是在 UBC（不列颠哥伦比亚大学），与基因组学研究人员一起工作。这些研究人员产生着非凡规模的海量测序数据，却将大量时间花费在将数据移动到需要的地方这一机械性工作上。永远在来回复制数据，管理多个不一致的副本。这是一个困扰着各行各业构建者的问题——从实验室里的科学家到训练机器学习模型的工程师——而这正是我们应该为客户解决的问题。
在这篇文章中，Andy 写了他的团队想出的解决方案：S3 Files。来之不易的经验教训，一些真正有趣的时刻，以及至少一次失败的尝试——试图为新的数据类型命名。这是一篇非常有趣的阅读，我相信你会喜欢的。
——W

第一部分：S3 的变迁

首先，一些植物学知识

事实证明，向日葵比人类要"滥情"得多。

大约十年前，就在加入 Amazon 之前，我刚结束第二次创业，回到 UBC 任教。我想探索一些我没有太多研究经验的领域，决定学习基因组学，特别是计算机系统与生物学家进行基因组学研究方式的交集。我最终花时间与 UBC 的植物学教授 Loren Rieseberg 一起工作，他研究向日葵 DNA——分析基因组以了解植物如何发展出让它们在干旱或盐碱土壤等恶劣环境中茁壮成长的特征。

植物学家关于"滥情"的笑话（开启这篇博客的那个笑话）是 Loren 的实验室如此有趣的原因之一。他们的解释是，人类 DNA 大约有 30 亿个碱基对，任何两个人在基因组水平上都有 99.9% 的相似性——我们所有的 DNA 都非常相似。但向日葵，作为花朵，而且完全不遵守一夫一妻制，不仅拥有更大的基因组（约 36 亿个碱基对），而且变异程度也大得多（个体间的遗传变异是人类的 10 倍）。

我当时的一位博士生 JS Legare 决定加入我的这次冒险，并在 Loren 的实验室做博士后，探索如何将这些工作负载迁移到云端。基因组分析是研究人员所谓的"突发并行"计算的一个例子。分析 DNA 可以通过大规模并行计算来完成，当你这样做时，它通常运行相对较短的时间。这意味着在实验室中使用本地硬件可能是一个糟糕的选择，因为当你需要进行快速分析时，你往往没有足够的计算能力，而当你不做主动工作时，你拥有的计算能力又处于闲置状态。我们的想法是探索使用 S3 和无服务器计算来并行运行成千上万个任务，这样研究人员就可以非常快速地运行复杂的分析，然后在完成后缩减到零。

生物学家在 Linux 环境下工作，使用一个名为 GATK4 的分析框架——一个与 Apache Spark 集成的基因组分析工具包。他们所有的数据都存储在一个共享的 NFS 文件服务器上。在连接到云端的过程中，JS 构建了一个他称之为"bunnies"的系统（另一个关于"滥情"的笑话），将分析打包到容器中并在 S3 上运行，这在速度、可重复性和通过并行化提高性能方面是一个真正的胜利。但一个突出的教训是存储边界的摩擦。

S3 在并行性、成本和持久性方面表现很好，但基因组研究人员使用的每一个工具都期望一个本地 Linux 文件系统。研究人员永远在来回复制数据，管理多个有时不一致的副本。这种数据摩擦——一边是 S3，另一边是文件系统，中间是手动复制管道——是我在此后几年中反复看到的情况。在媒体和娱乐业，在机器学习的预训练中，在芯片设计中，在科学计算中。不同的工具以不同的方式编写来访问数据，当位于我们数据前面的 API 成为一种摩擦源，使工作变得更加困难时，这真的很糟糕。

Agent 放大数据摩擦

我们都很清楚，也许还有点震惊，代理工具正在改变今天的软件开发。Agent 在编写代码方面相当不错，而且它们的进步速度足以让我们都花相当多的时间思考这一切到底意味着什么（甚至 Werner 也是）。但有一件事似乎是真的：代理开发已经深刻地改变了构建应用程序的成本。成本不仅体现在金钱上，体现在时间上，特别是在编写可行代码的技能上。这最后一部分是我最近发现最令人兴奋的，因为在我们拥有软件的大部分时间里，成功的应用程序总是涉及结合两个往往不连贯的技能组合：一方面是正在编写的应用程序领域的技能，如基因组学、金融或设计，另一方面是实际编写代码的技能。在很多方面，Agent 正在说明编写软件的准入门槛一直高得令人望而却步，突然间允许更大范围的人编写应用程序——那些在正在编写的应用程序领域拥有深厚技能的人，而不是在编写它们的机制方面。

当我们发现自己处于这样一个位置——应用程序比以往任何时候都编写得更快、更具实验性、更加多样化时，从想法到运行代码的周期时间正在急剧压缩。随着构建应用程序成本的崩溃，以及我们构建的每个应用程序都可以作为下一个应用程序的参考，代码/数据的划分似乎变得比以往任何时候都更有意义。我们正在进入一个应用程序来来去去的时代，而一如既往，数据比所有应用程序都更长久。有效存储系统的作用一直不仅仅是安全存储数据，还包括帮助抽象和解耦数据与单个应用程序。随着应用程序开发步伐的加快，存储的这一特性变得比以往任何时候都更加重要，因为数据越容易附加和使用，我们就越能玩转、构建和探索从中受益的新方法。

S3 作为你数据的管理者

在过去的几年里，S3 团队一直非常关注这最后一点。我们一直在仔细观察 S3 中数据访问方式不够简单的情况——就像 Loren 实验室的生物学家的例子，他们不得不构建脚本来复制数据，以便它位于正确的位置供他们的工具使用——我们开始更广泛地查看客户发现使用存储分散了他们使用数据的注意力的情况。我们在这里得到的第一个教训是关于结构化数据。S3 存储了 EB 级的 parquet 数据，仅该格式平均每秒就有超过 2500 万个请求。其中很多要么是普通的 parquet，要么结构化为 Hive 表。很明显，人们想用这些数据做更多的事情。开放表格格式，特别是 Apache Iceberg，正在成为功能更丰富的表格抽象，允许插入和变更、模式更改和表格快照。虽然 Iceberg 显然有助于提升 S3 上表格数据的抽象水平，但它仍然带有一些尖锐的边缘，因为它必须严格通过对象 API 来呈现表格。

随着 Iceberg 开始流行，大规模采用它的客户告诉我们，管理安全策略很困难，他们不想管理表格维护和压缩，他们希望使用表格数据更容易。此外，Iceberg 和开放表格格式（OTF）的大量工作专门针对 Spark 驱动。虽然 Spark 作为分析引擎非常重要，但人们将数据存储在 S3 中是因为他们希望能够使用任何他们想要的工具，甚至（尤其是！）那些还不存在的工具。因此，在 2024 年的 re:Invent 上，我们推出了 S3 Tables，作为一个托管的、一等表格原语，可以作为结构化数据的构建块。S3 Tables 以 Iceberg 格式存储数据，但增加了护栏以保护数据完整性和持久性。它使压缩自动化，增加了对跨区域表格复制的支持，并继续完善和扩展表格应该是一等数据原语的概念，与对象一起作为构建应用程序的方式。今天，我们在 S3 Tables 中存储了超过 200 万个表格，看到各种卓越的应用程序建立在它们之上。

大约在同一时间，我们开始与 S3 客户就相似性搜索和向量索引进行大量对话。过去几年 AI 的进步确实为各种存储数据的向量索引创造了机会和需求。机会来自先进的嵌入模型，它们在提供语义搜索的能力上引入了阶跃式变化。突然间，拥有大型档案媒体集合的客户，如历史体育镜头，可以构建向量索引并实时搜索特定球员达阵得分的剪辑，并立即获得一个片段集合，组装成一个精彩集锦，可用于现场直播。语义相关搜索的同一特性对于 RAG 和在未训练过的数据上应用模型同样有价值。

当客户开始构建和操作其数据上的向量索引时，他们开始强调一个略微不同的数据摩擦源。强大的向量数据库已经存在，向量作为功能已经快速进入现有数据库如 Postgres。但这些系统将索引存储在内存或 SSD 上，作为带有活动索引的计算集群运行。这是连续低延迟搜索设施的正确模型，但如果你从存储角度来看待你的数据，它就不那么有用了。客户发现，特别是在基于文本的数据（如代码或 PDF）上，向量本身的字节数往往比被索引的数据还多，存储在价格昂贵许多倍的介质上。

因此，就像团队在 S3 Tables 的结构化数据工作一样，在上次 re:Invent 上，我们推出了 S3 Vectors 作为 S3 原生向量索引的新数据类型。S3 Vectors 对存储向量采取了非常 S3 的方式，其设计锚定在与 S3 对象非常相似的性能、成本和持久性配置。但可能最重要的是，S3 Vectors 设计为完全弹性，这意味着你可以快速创建一个只有几百条记录的索引，并随着时间扩展到数十亿条记录。S3 Vector 最大的优势在于拥有一个始终可用的 API 端点来支持相似性搜索索引的纯粹简单性。就像对象和表格一样，它是另一个你可以在应用程序开发中直接使用的数据原语。

现在……S3 Files

今天，我们推出 S3 Files，一个将 Amazon Elastic File System（EFS）集成到 S3 中的新功能，允许任何现有的 S3 数据作为网络附加文件系统直接访问。

关于文件的故事实际上比 Tables 或 Vectors 的工作更长，甚至更有趣，因为文件原来是一种复杂而棘手的数据类型，难以与对象存储干净地集成。我们实际上在推出 S3 Tables 之前就开始研究文件的想法，作为 EFS 和 S3 团队的联合努力，但让我们先把这个放一放。

正如我在分析向日葵 DNA 的基因组学示例中描述的那样，有大量现有软件通过文件系统 API 处理数据——数据科学工具、构建系统、日志处理器、配置管理和训练管道。如果你观察代理编码工具处理数据的方式，它们非常快地伸手去使用丰富的 Unix 工具范围，直接处理本地文件系统中的数据。在 S3 中处理数据意味着它们必须深入推理，主动列出 S3 中的文件，将它们传输到本地磁盘，然后操作这些本地副本。这显然比代理用例更广泛，对于今天在其作业中使用本地文件系统的每个客户应用程序都是如此。在 S3 上原生支持文件使所有这些数据立即可访问——并最终更有价值。你不必从 S3 复制数据来使用 pandas，或指向训练作业，或使用设计工具与其交互。

有了 S3 Files，你会得到一个非常简单的东西。你现在可以在 EC2 VM、容器或 Lambda 函数内挂载任何 S3 存储桶或前缀，并通过文件系统访问该数据。如果你进行更改，你的更改将传播回 S3。因此，你可以将对象作为文件处理，将文件作为对象处理。

这就是故事变得有趣的地方，因为正如我们在试图为客户简化事情时经常学到的那样，使事情简单往往是你着手做的更复杂的事情之一。

第二部分：S3 Files 的设计

构建者讨厌他们必须尽早决定数据是驻留在文件系统还是对象存储中，并从那时起承受其后果这一事实。有了这个决定，他们基本上是在选择不仅是现在，而且在遥远的未来如何与他们的数据交互，如果他们弄错了，他们要么进行迁移，要么构建一层自动化来复制数据。

最初，我们的想法基本上是将 EFS 和 S3 放进一个大锅里，炖一会儿，我们就会得到两全其美。我们甚至称项目的早期版本为"EFS3"（我很高兴我们没有保留这个名字！）。但事情很快变得棘手起来。每次我们坐下来研究设计时，我们都会发现困难的技术挑战和艰难的决定。在这些决定中的每一个中，文件或对象的数据呈现必须在设计中放弃一些东西，使其不那么好。团队中的一位工程师将其描述为"难以接受的妥协之战"。我们绝不是第一批发现将文件和对象融合到单一存储系统中有多困难的存储人员，但我们也敏锐地意识到，没有解决方案让构建者多么沮丧。

我们决心找到一条出路，所以我们做了面对真正困难的技术设计问题时唯一明智的事情：我们将一群最资深的工程师锁在一个房间里，说在他们拿出一个他们都喜欢的计划之前不会让他们出来。

激烈而充满争议的讨论随之而来。一次又一次。又一次。最终我们放弃了。我们就是无法得到一个不让某人（在大多数情况下真的是每个人）对设计不满意的解决方案。

这里稍微说一下：我可能在关于把人锁在房间里的评论上采取了一些戏剧性的自由。Amazon 会议室的门上没有锁。但在这个问题上我要说清楚：我经常发现，当我们让聪明、有激情、技术观点不同的人聚集在白板前，在几天的时间里真正深入探讨时，我们在非常困难的设计问题上取得最快和最有建设性的进展。这不是一个惊天动地的观察，但在试图通过一小时一小时的视频会议谈论大型困难问题时，往往很容易忘记这一点。参与这些讨论的工程师深入理解文件和对象工作负载以及它们的不同之处，所以这些讨论深入、有时激烈，而且绝对令人着迷。尽管如此，我们仍然无法得到我们喜欢的设计。这真的令人沮丧。

这是 2024 年圣诞节前后。进入假期前，团队改变了方向。他们梳理了设计文档和讨论笔记，开始列举如果我们想要将文件和对象接口作为统一的单一系统呈现，我们需要接受的所有具体设计妥协和行为。我们都看了一遍，同意这不是两全其美，这是最低公分母，我们都能想到双方的示例工作负载会以令人惊讶的、往往微妙的、总是令人沮丧的方式崩溃。

我认为对我来说真正突出这个问题的例子是关于对象和文件作为数据原语实际上有何不同的顶层语义和体验。这里有一个极其简单的特征：文件是操作系统构造。它们存在于存储上，在断电时持久存在，但当它们被使用时，它们作为表示数据的方式极其丰富，以至于它们经常被用作跨线程、进程和应用程序通信的方式。文件的 API 构建为支持这样的想法：我可以就地更新数据库中的记录，或追加数据到日志，而你可以并发访问该文件并几乎即时看到我的更改，到文件的任意子区域。有一套丰富的 OS 功能，如 mmap()，它加倍强调文件作为共享持久数据，可以以非常细的粒度变更，就像它是一组内存数据结构一样。

现在如果我们转向对象世界，在别人访问对象时写入对象中间的想法或多或少是亵渎。对象的不变性是嵌入 API 和应用程序的假设。工具会下载并验证内容哈希，它们会使用对象版本控制来保留旧副本。最值得注意的是，它们经常构建完全基于与整个对象创建相关联的通知的复杂工作流程。最后这件事在我开始从事 S3 工作时让我惊讶，它实际上非常酷。像 S3 跨区域复制（CRR）这样的系统基于对象创建或覆盖时发生的通知复制数据，这些通知被期望具有至少一次语义，以确保我们永远不会错过对象的复制。客户使用类似的管道触发日志处理、图像转码和各种其他东西——这是对象应用程序设计的一个非常流行的模式。事实上，通知是 S3 子系统的一个例子，让我对我得以工作的存储系统的规模感到惊叹：S3 每天仅向处理新对象的无服务器事件监听器发送超过 3000 亿个事件通知！

我们开始意识到的是，文件和对象之间实际上存在相当深刻的边界。文件交互是敏捷的、通常变更繁重的、语义丰富的。另一方面，对象带有一套相对集中和狭窄的语义；我们意识到，我们需要关注的正是这个分隔它们的边界，而不是试图隐藏它，边界本身是我们需要构建的功能。

暂存和提交

假期结束后，我们开始（好吧，不完全是锁）把人关在房间里，但这次的观点是文件和对象之间的边界实际上不必是不可见的。这一次，团队开始从讨论中走出来，看起来快乐多了。

第一个决定是，我们将把 S3 上的一流文件访问视为处理数据的表现层。我们将允许客户在存储桶或前缀上定义 S3 挂载，在底层，该挂载将附加一个 EFS 命名空间来镜像 S3 的元数据。我们将使两个层级之间的数据传输和一致性成为我们设计的绝对核心部分。我们开始将此描述为"暂存和提交"，这是我们从 git 等版本控制系统中借用的术语——变更将能够在 EFS 中累积，然后集体下推到 S3——数据如何以及何时跨边界传输的具体细节将作为系统的一部分发布，对客户清晰可见，并且我们可以继续发展和改进的东西，作为随时间推移的程序化原语。（我将在最后更多地谈论这一点，因为团队在这个表面上有更多兴奋要做的事情）。

明确文件和对象呈现之间的边界是我完全没有预料到的，当团队开始研究 S3 Files 时，这是我真的开始喜欢这个设计的一点。现在还早，我们有充足的演进空间，但我认为团队都觉得这为我们在与构建者需求合作改进和演进方面奠定了基础，而不是被困在那些难以接受的妥协后面。

尚未走出困境

决定这个暂存和提交的事情是提供一些边界和关注点分离的设计决策之一。它给了我们一个清晰的结构，但它并没有让难题消失。团队仍然必须导航文件和对象语义、性能和一致性之间的真正权衡。让我通过几个例子来展示这两个抽象实际上有多微妙，以及团队如何处理这些决定。

一致性和原子性

S3 读者通常假设完整的对象更新、通知，以及在许多情况下访问历史版本。文件系统具有细粒度变更，但它们也有重要的一致性和原子性技巧。许多应用程序依赖于进行原子文件重命名以使大更改一次性可见的能力。他们对目录移动做同样的事情。S3 条件对第一件事有一点帮助，但不是完全匹配，对第二件事没有 S3 对应物。因此，如上所述，分离层级允许这些模态在具有相同数据单一视图的并行系统中共存。你可以随心所欲地变更和重命名文件，在稍后的时间点，它将作为整体写入 S3。

授权

授权同样棘手。S3 和文件系统以非常不同的方式思考授权。S3 支持作用域限于键前缀的 IAM 策略——你可以说"拒绝 /private/ 下任何内容的 GetObject"。实际上，你可以根据网络或请求本身的属性等进一步约束这些权限。IAM 策略极其丰富，而且评估起来比文件权限昂贵得多。文件系统花费多年时间将权限检查之类的事情移出数据路径，通常预先评估，然后使用句柄进行持久的未来访问。文件作为包装授权策略的实体也有点奇怪，因为文件的权限驻留在其 inode 中。硬链接允许你对同一文件拥有多个 inode，你还需要考虑决定你是否能首先访问文件的目录权限。除非你有句柄，在这种情况下，即使它被重命名、移动，甚至经常被删除，这都无关紧要。

这里有更多的复杂性，呃，丰富性——特别是围绕用户和组身份等主题——但通过转向明确的边界，团队避免了必须在每个对象上共同表示两种类型的权限。相反，权限可以在挂载本身上指定（网络文件系统用户熟悉的领域）并在文件系统内强制执行，在两个世界之间应用特定的映射。

这个设计有另一个优势。它保留了 S3 上的 IAM 策略作为后备。如果需要更改数据边界，你始终可以在 S3 层禁用访问，同时将授权委托给每个挂载内的文件层。它为未来我们可能想要探索同一数据上多个不同挂载的情况敞开了大门。

命名空间语义的可怕不一致

如果你熟悉文件和对象系统，思考文件和对象命名行为大不相同的情况并不难。当你开始坐下来真正深入研究时，事情几乎变得可笑地荒凉。文件系统有一等路径分隔符——通常是正斜杠（"/"）字符。S3 也有这些，但它们真的只是一个建议。实际上，S3 的 LIST 命令允许你指定任何你想解析为路径分隔符的东西，有少数客户构建了卓越的多维命名结构，在同一路径中嵌入多个不同的分隔符，并根据他们想要如何组织结果向 LIST 传递不同的分隔符。

这是另一个简单而恼人的例子：因为 S3 没有目录，你可以拥有以同一个斜杠结尾的对象。也就是说，你可以拥有一个看起来像目录但实际是文件的东西。大约 20 分钟，团队认为这是一个很酷的功能，称它们为"文件录"（filerectories）。谢天谢地我们没有保留那个名字。

有几十个这样的差异，我们仔细考虑了限制为单一共同结构或只是将自己固定在一侧或另一侧。在所有这些路径上，我们意识到我们要破坏应用程序内部关于命名的假设。

我们决定倾向于边界，允许双方坚持其现有的命名约定和语义。当创建的对象或文件无法跨边界移动时，我们决定（哇，这真的是很多激烈的讨论）我们只是不移动它们。相反，我们会发出一个事件，允许客户监控并在必要时采取行动。这显然是将复杂性下放给开发者的例子，但我认为这也是这样做是正确选择的深刻好例子，因为我们选择不在它们已经期望运行的域中失败，我们正在构建一个边界，承认绝大多数在两种情况下确实有效的路径名，我们正在构建一种机制来检测和纠正出现的问题。

性能体验

团队花大量时间谈论的最后一个差异领域是性能，特别是命名空间交互的性能和请求延迟。文件和对象命名空间针对非常不同的事物进行优化。在文件系统中，存在大量数据依赖的元数据访问。访问文件意味着也访问（在某些情况下更新）目录记录。还有许多操作最终沿路径遍历所有目录记录。因此，快速文件系统命名空间——即使是大型分布式命名空间——倾向于将目录的所有元数据共置在单个主机上，以便这些交互尽可能快。对象命名空间完全平坦，倾向于针对非常高度并行的点查询和更新进行优化。在 S3 中有许多情况，单个"目录"中有数十亿个对象，并被数十万客户端并行访问。

当我们查看刚才描述的一系列挑战时，我们花了很多时间谈论采用。S3 已经 20 岁了，我们希望现有 S3 客户可以立即在他们自己的数据上使用的解决方案，而不是意味着迁移到全新东西的解决方案。有大量现有存储桶服务于依赖 S3 对象语义完全按照文档工作方式的应用程序。我们不愿意引入可能破坏这些应用程序的微妙新行为。

事实证明，很少有应用程序在完全相同的时间实例在同一数据上同时使用文件和对象接口。更常见的模式是多阶段的。数据处理管道在一个阶段使用文件系统工具生成输出，在下一阶段被基于对象的应用程序消费。或者客户想要对通过文件系统积极修改的数据快照运行分析查询。

我们意识到，不必融合文件和对象语义来解决数据孤岛问题。他们需要的是将同一数据放在一个地方，为每种访问模式提供正确的视图。一个提供完整 NFS 关闭到打开一致性的文件视图。一个提供完整 S3 原子 PUT 强一致性的对象视图。以及一个保持它们连接的同步层。

所以我们发布了它

所有那些争论——团队的"难以接受的妥协"清单，关于文件录的激烈偶尔荒凉的讨论——结果证明正是我们需要做的工作。我认为团队都觉得设计因为经历过这些而变得更好。S3 Files 让你在 EC2 实例、容器或 Lambda 函数上将任何 S3 存储桶或前缀挂载为文件系统。在底层，它由 EFS 支持，提供你的工具已经期望的文件体验。NFS 语义、目录操作、权限。从你的应用程序角度来看，它是一个挂载的目录。从 S3 的角度来看，数据是存储桶中的对象。

它的工作方式值得快速过一遍。当你首次访问目录时，S3 Files 从 S3 导入元数据并填充同步视图。对于小于 128 KB 的文件，它也会拉取数据本身。对于较大的文件，只有元数据过来，数据在你实际读取时从 S3 获取。这种延迟水分化很重要，因为这意味着你可以挂载一个包含数百万对象的存储桶并立即开始工作。这个"立即开始工作"部分是一个简单体验的好例子，实际上在底层非常复杂——能够挂载并立即将 S3 中的对象作为文件使用是该功能明显和自然的期望，必须等待几分钟或几小时才能填充元数据的文件视图会相当令人沮丧。但在底层，S3 Files 需要扫描 S3 元数据并为其填充文件优化的命名空间，团队能够非常快地做到这一点，作为保留简单和非常敏捷客户体验的后台操作。

当你创建或修改文件时，变更被聚合并作为单个 PUT 每大约 60 秒提交回 S3。同步双向运行，所以当其他应用程序修改存储桶中的对象时，S3 Files 自动发现这些修改并自动在文件系统视图中反映它们。如果曾经有文件同时从两个地方修改的冲突，S3 是事实来源，文件系统版本移动到 lost+found 目录，并有 CloudWatch 指标标识该事件。30 天内未访问的文件数据从文件系统视图中驱逐，但不会从 S3 中删除，因此存储成本与你的活动工作集成比例。

在团队构建系统时发生了许多更小、真正有趣的工作。我认为真正酷的改进之一是我们所说的"读取旁路"。对于高吞吐量顺序读取，读取旁路自动重新路由读取数据路径，不使用传统 NFS 访问，而是直接对 S3 本身执行并行 GET 请求，这种方法实现每个客户端 3 GB/s（还有进一步改进空间）并跨多个客户端扩展到每秒太比特。对于那些感兴趣的人，我们的技术文档中有更多细节（这很有趣的阅读）。

我真正开始欣赏这个设计的一点是它对自己的边缘有多诚实。文件和对象域之间的明确边界不是我们在掩盖的限制。它是让双方保持不妥协的东西。也就是说，有些地方我们知道我们还有工作要做。重命名很昂贵，因为 S3 没有原生重命名操作，所以重命名目录意味着复制和删除该前缀下的每个对象。正是出于这个原因，当挂载覆盖超过 5000 万个对象时我们会警告你。明确的提交控制在发布时还不存在；60 秒窗口对大多数工作负载有效，但我们知道它不足以满足每个人的需求。还有一些对象键根本无法表示为有效的 POSIX 文件名，所以它们不会出现在文件系统视图中。我们已经在客户 beta 测试大约九个月，这些是我们学到的并继续与早期客户演进和迭代的东西。我们宁愿清楚说明它们也不愿假装它们不存在。

文件和向日葵

当我们在 UBC 与 Loren 的实验室合作时，JS 花了大量时间构建缓存和命名层——不是在做生物学，而是在编写基础设施，在数据居住的地方和工具期望它所在的地方之间穿梭数据。这种摩擦真的很突出，现在回顾它，我认为我们不断吸取的教训——在那个实验室，然后一次又一次当 S3 团队研究 Tables、Vectors 和现在的 Files 时——是不同的数据工作方式不是要坍塌的问题。它们是要服务的现实。Loren 实验室的向日葵在变异上茁壮成长，数据访问模式也是如此。

我发现 S3 Files 最令人兴奋的是我一开始完全没预料到的：文件和对象之间的明确边界结果证明是设计的最佳部分。我们花了几个月试图让它消失，当我们最终接受它作为系统的一等元素时，一切都变得更好了。暂存和提交给了我们可以继续演进的表面——对数据何时以及如何跨边界传输的更多控制、与管道和工作流程的更丰富集成——它让我们在不妥协任何一方的情况下做到这一点。

20 年前，S3 作为对象存储开始。在过去的几年里，有了 Tables、Vectors 和现在的 Files，它变成了更广泛的东西。一个数据持久存在并可以以对当前工作有意义的方式工作的地方。我们的目标是让存储系统不阻碍你的工作，而不是你必须绕过的东西。我们还远未完成，但我对我们前进的方向真的很兴奋。

正如 Werner 所说，"现在，去构建吧！"
