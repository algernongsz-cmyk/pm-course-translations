如何为给你的 AI Agent 系统访问权限提出合理依据

Dawn Perrott · AI & Automation · 2026年6月11日 · 阅读时间：7分钟

如果不能访问你的后端系统，你的 AI Agent 只能回答问题，却无法采取行动。

客户要求更改付款计划，Agent 会给出清晰的解释，但客服代表仍然需要介入并进行更改。其他人需要更新账户，同样的事情再次发生。Agent 知道答案，只是没有行动的能力。

回答查询和解决查询之间的差距让你的团队继续处理 Agent 本可以承担的请求。弥合这一差距意味着将其连接到工作发生的系统，如你的 CRM、计费平台或订单管理工具。这通常是工程部门的要求，大多数支持团队很难将其优先级提高。以下是如何提出这个依据。

系统访问权限能带来什么改变

强大的知识管理系统使你的 Agent 能够解决很多查询。但当客户需要做某些事情时，它能回答的和能行动的之间有一条清晰的界线：

没有系统访问权限 有系统访问权限
你的 Agent 告诉客户如何提交损坏订单索赔。 你的 Agent 处理索赔，在数据库中检查订单状态，并确认替换 —— 所有这些都在一次对话中完成。
你的 Agent 告诉客户登录查看订阅续订日期。 你的 Agent 实时查找续订日期和订阅状态，并立即给客户答案 —— 无需登录。

从回答到行动的转变是 AI 优先支持经济学的改变。每个 Agent 能够端到端解决的查询都是不再落在你团队身上的工作，这个区别证明了工程要求的合理性。

数据显示了什么

根据我们的2026年客户服务转型报告，87%的成熟 AI 部署团队 —— 即 AI 已集成到支持运营并在规模上工作 —— 报告指标改善，而总体为62%。虽然82%的高级领导表示他们的团队在过去一年投资了 AI，但只有10%表示他们已达到成熟部署阶段。

区分采用和成熟的大部分因素是集成。Agent 擅长回答问题，但如果没有系统访问权限，它就无法完成工作。

我们自己的支持团队直接测试了这一点。我们将四个最高工作量的工作流程作为固定的脚本化工作流程运行 —— 在 Fin 中称为 Tasks。它们适用于简单的线性流程，但无法处理复杂性。当我们将其重建为 Procedures（具有真正系统访问权限的工作流程）时，结果并不一致。这正是重点所在。Procedures 在需要判断、分支逻辑、实时数据或更好交接的工作中创造了最大的提升。

流程 Task Procedure 变化
Bounce list（退信列表） 9.3% 79.9% +70.6 pp
Report a bug（报告bug） 9.2% 66.5% +57.3 pp

数据反映截至2026年5月的过去12个月

每个流程因不同原因而改进。例如：

  • Bounce list 管理因投递失败而被阻止接收未来消息的电子邮件地址。它需要判断，包含多步逻辑、错误恢复和动态分支 —— 这些是 Task 永远无法处理的事情。
  • Bug reporting 仍然需要人工交接，但交接质量有所提高。团队成员收到预先分类的工单，GitHub issue 匹配已经浮现，正确的 URL 已提取，模拟访问已请求。
  • Messenger installation 几乎没有变化，因为不需要。这已经是一个 Tasks 处理得很好的简单线性工作流程。

并非每个工作流程都需要更深层次的集成，但需要的那些是最大收益所在。

如何界定请求范围

Agent 集成的最强内部案例始于一个紧密范围的请求。

你最好的第一个候选是高量、可重复、与明确系统所有者相关，并且有现有 API 或现实路径。查看 Agent 的分析数据中的模式：哪里在解释流程而不是完成流程？哪里被告知登录、检查其他系统或等待人工？这些是你的起点。

用简单的语言逐步映射工作流程。标记 Agent 需要读取数据的地方和需要采取行动的地方。定义每个系统所需的最小字段集。请求越集中，越容易批准。

如果你使用 Fin，Recommendations 仪表板直接浮现这些见解 —— 按对话量优先级排序 —— 并包括 API 需求和所需数据、示例架构和每个的工作量评级。

将此包含在工程资源的案例中，这样你的请求已经界定范围，更容易评估。

分阶段思考

最成功的团队会随着时间增加集成，而不是试图一次性连接所有东西：

阶段 1：无需集成

使用 Agent 进行引导式故障排除、分类、策略检查和路由逻辑。这不需要工程工作，可以帮助你识别哪些工作流程最受益于系统访问权限。

阶段 2：只读访问

将 Agent 连接到一个系统，以便它可以查找信息，如订单状态或订阅详情。这通常是第一个工程请求 —— 一个工作流程、一组小字段、无写权限。

阶段 3：写入操作

让 Agent 在系统中采取行动，如退款、取消订阅或更新记录。这是更深层次的集成，通常在团队对早期阶段建立信心后进行。

如何保持势头

当你在集成案例中工作时，工程团队可能会对容量、系统访问范围以及如何与现有路线图优先级排序提出问题。

以下是如何解决这些问题：

1. 定义容量

你不需要预先承诺大量工作。从一个针对一个重复的高量工作流程的狭窄试点开始。单个集成的工程投入通常比团队假设的要小。如果你使用 Fin，Operator 可以从简单的语言描述起草初始工作流程，这意味着需求上的反复更少。

2. 界定系统访问范围

从小规模开始并共同定义边界。将集成范围限定到特定端点和一组小的批准字段。只读访问通常是正确的起点，这意味着没有写权限，没有意外更改的风险。

3. 解决 API 就绪问题

完全构建的 API 不必先于一切。大多数 Agent 支持模拟响应，让你使用测试场景预先构建和验证工作流程逻辑。如果你使用 Fin，并且集成 —— 使用 Data Connectors 配置 —— 仍然需要几个 sprint，那么人工循环步骤可以充当时替代，团队成员可以手动完成步骤，同时你收集整个工作流程影响的数据。这些数据为优先真正的集成提供了依据。

4. 纳入工程路线图

如果将 Agent 与后端系统集成不在工程团队本季度的路线图上，利用这段时间做好准备。映射流程、记录所需字段、定义成功指标。当容量开放时，一个完全范围化的请求和清晰的预期影响比还需要定义的请求更容易安排。你现在做的准备工作使后来的工程对话更短。

从窄开始，然后扩展

第一次集成改变了内部对话。一旦领导层看到真实工作流程的解决率提高，工程团队看到集成实际涉及什么，第二个请求就从不同的基线开始。

你的 Agent 端到端解决的每个工作流程都是少一个落在支持代表身上的任务。在规模上，这意味着经验丰富的支持团队将时间花在真正需要人工判断的工作上。

对于更深层次集成的最强案例是你的团队仍在做但 Agent 可以处理的工作，以及继续没有它的成本。

从系统集成中获得最大价值的团队不会一次性要求所有东西。他们从一个工作流程开始，衡量结果，并使用该证明为下一步提出依据。
