对话设计：如何让你的 AI Agent 像团队一样沟通

原文作者：Fred Walton | 发布日期：Jun 18, 2026 | 阅读时间：6 分钟

分类：AI & Automation

如果你的团队里没有人负责训练 Agent 如何沟通，当它与客户对话时，听起来就会像一个 LLM——因为它确实就是一个 LLM。

对话设计（Conversation design）是 AI-first 支持团队中一项新兴的学科，专门用来解决这个问题。对话设计师负责 Agent 的沟通方式：语气、结构、详细程度、客户体验、交接和升级流程。

如果没有专门的人来定义 Agent 应该遵循的沟通指导原则，它就会自己做出决定。这可能导致：当简短回答就够了时，它却提供了过多细节；当客户感到沮丧时，它却用平淡的语调回复；或者在应该交接时迟迟没有触发交接。

这些问题的成本是可以衡量的。收到尴尬结构化回答的客户不太可能信任这些答案——即使答案本身是准确的——所以他们会升级到人类客服，只是为了听同样的事情以不同的方式说出来。有些人会完全跳过 Agent。当 Agent 确实进行交接时，糟糕的过渡意味着人类客服继承了一个已经感到沮丧的客户。每一个这样的结果都是可以避免的，而对话设计正是确保这些情况不会发生的学科。

我们在 Fin 亲眼见证了这一点。我们 A/B 测试了两种开场消息：一种温暖且对话式，另一种是我们较旧的默认版本。对话式的问候将 CSAT 从 72.8% 提升到了 78.4%。一个单一的对话设计变更，应用到客户首先看到的内容上，就产生了可衡量的差异。

对话设计涵盖的内容

这个角色涵盖五个不同的领域，每个领域塑造客户体验的不同部分：

领域 描述
语气和个性 声音、详细程度、Agent 听起来有多正式或随意，以及是否根据情况变化。
响应结构 Agent 是否将详细程度与客户的问题相匹配。
交接逻辑 何时升级、如何沟通过渡、以及传递什么上下文。
交互流程 对话如何通过提问、回答、解决或交接进行。
响应质量 答案是否感觉清晰、有帮助、符合品牌，即使技术上正确。

如何付诸实践

从对话应该感觉如何开始

在调整个别响应之前，先定义声音。用一段话写下你希望 Agent 听起来如何。你不需要完整的品牌指南，只需要一个可以在做语气决定时回来参考的基准。

不同的对话类型可能需要不同的语调。一个账户被锁定的客户需要直接和速度，而一个探索新功能的人可能更喜欢更多上下文。声音保持一致，但语调应该适应。

仔细设计交接

从 Agent 到客服的过渡是摩擦最高的时刻之一。客户不应该需要重新解释他们的问题。客服应该收到完整的对话历史、问题背后的上下文、Agent 已经做了什么、以及为什么发生升级。

Agent 沟通交接的方式也很重要。"让我连接一个可以帮助这个问题的团队成员"感觉不同于无声的切换。

设计一个安全机制也是必不可少的。如果 Agent 不能干净地解决对话，你需要一个后备方法，仍然给客户一个平滑的交接体验。客户此时可能对 AI 感到沮丧，但一个处理得当的过渡可以扭转这种情况。

不要忘记跟进

跟进需要和交接同样的关注。如果有人在对话中途放弃了——无论是你的 Agent 还是客服——你如何重新联系以确保他们得到了需要的帮助？大多数团队不会考虑这一点，而客户会注意到。

知道 Agent 应该何时停止说话

最常见的对话设计错误之一是过度解释。Agent 可以访问大量信息，在没有指导的情况下，它很容易给出比客户需要更多的细节。

Agent 应该将详细程度与客户的问题相匹配。一个问如何重置密码的人不需要三段话。一个问复杂集成的客户可能需要。如果有更多可以分享的，它应该提供而不是一次全部给出。

为客户正在进行的对话设计

客户不遵循脚本。他们会在对话中途改变方向或提出与他们原始问题无关的后续问题。

Agent 需要处理这些过渡，而不是强迫客户回到固定的流程。当 Agent 在客户已经转向之后仍然试图解决原始问题时，感觉就像与一个不在倾听的人对话。

考虑相同的流程是否应该适用于不同的渠道，以及不同的客户细分是否需要不同的体验。

保持你的指令简短

最大的实际挑战之一是过度指导 Agent。每当新的边缘情况出现时，团队就添加规则。很快，LLM 在响应之前需要处理段落式的指令。

我在 Fin 看到这发生了，我也从其他团队听到同样的情况。本能总是添加更多，但纪律是知道何时停止。

我的规则：如果是关于内容或信息，它属于知识库。如果是关于语气或如何处理特定情况，它属于你的 Agent 指令。"在定价上要直接"比一段解释你定价沟通策略背后哲学的话做得更多。

如果你正在使用 Fin，大部分工作发生在 Guidance 中。这是对话设计成形的地方，帮助你定义 Agent 应该如何听起来、应该说多少、以及在不同情况下应该如何响应。

无需专门招聘如何开始

大多数团队不会在第一天就招聘专门的对话设计师——这没问题。但某人需要负责 Agent 如何沟通，即使这是现有角色的一部分。

对话设计通常从支持运营或知识管理内部开始。团队中某人开始关注 Agent 听起来如何。随着时间的推移，当你的 Agent 处理更多对话时，这成为一个正式责任，最终成为一个专门角色。

从哪里开始

1. 指定负责人

你需要某人负责 Agent 如何沟通。这不需要是新招聘，但需要明确指定。

2. 选择一个效果不佳的对话类型

看看那些 Agent 正确回答但客户仍然升级或留下负面反馈的对话。从那里开始。

如果你正在使用 Fin，CX Score 可以帮助你发现这些。它显示哪些主题和对话类型得分低，以及这些分数背后的原因，这样你可以看到问题是否是答案质量、客户努力，还是其他原因。

3. 审计你的 Agent 指令

如果它们已经超出几个集中的规则，修剪它们。将内容移到知识库，保持指令专注于行为。

4. 修复你最差的交接

走查几个 Agent 升级到人类的对话。客户必须重复自己吗？客服有足够的上下文吗？先重新设计那个单一的过渡。

小步骤会累积

每一个这些改进的影响都会累积。一个温暖的开场消息提升了我们的 CSAT，同时修剪指令使响应更清晰。设计更好的交接意味着客服不再继承沮丧的客户。

这些变更都不需要新知识；它们需要某人关注对话本身。
