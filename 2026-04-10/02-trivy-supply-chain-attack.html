作者：VaultProof

攻击剖析

2026 年 3 月 19 日，Aqua Security 的 Trivy——世界上使用最广泛的漏洞扫描器之一——遭到入侵。攻击者直接向官方发布二进制文件注入了凭据窃取逻辑。

该攻击载荷非常复杂：扫描看起来正常完成并通过。凭据泄露在合法功能旁边静默运行。团队没有任何迹象表明出了问题。

这次攻击不需要在你的代码中找到漏洞。它利用了你的 CI/CD 管道运行具有访问你的环境权限的工具——而你的 API Key 就以明文字符串的形式存在于该环境中。

这就是使传统密钥管理不足的供应链攻击模型：如果密钥在你的运行时环境的任何地方作为明文字符串存在，受损的工具就可以找到并泄露它。

入口点：攻击者入侵 Trivy 发布版本

利用可变的 Git 标签和自声明的提交身份，向官方 v0.69.4 发布二进制文件注入恶意软件。

传播：GitHub Actions 接收攻击载荷

trivy-action 和 setup-trivy GitHub Actions 同时遭到入侵。数百万个 CI/CD 管道现在运行恶意代码。

泄露：从运行时环境窃取凭据

恶意载荷从环境变量访问明文 API Key——这正是每个密钥管理器检索后放置它们的地方。密钥被发送到攻击者的 C2 服务器。

VaultProof 在哪里打破链条：不存在明文密钥可供窃取

使用 VaultProof，完整的 API Key 从不存在于 CI/CD 环境中。只存在加密份额——对攻击者来说单独使用是无用的。没有东西可窃取。

盲点：为什么你的密钥管理器没有帮助

2026 年 3 月可用的每个密钥管理器——Vault、AWS Secrets Manager、Doppler、Infisical——都遵循相同的检索模型。你将密钥加密存储。你的 CI/CD 管道在运行时通过 API 检索它。密钥变成明文环境变量，你的工具可以读取它。

这是有意为之的。这是这些工具的设计方式。它们保护静态的密钥——而不是使用中的密钥。

Trivy 恶意软件不需要找到漏洞。它只是读取已经存在的内容。你的密钥管理器完全按照设计做了它应该做的事情——攻击者仍然得到了密钥。

解决方案：什么本可以阻止这一切

针对针对凭据的供应链攻击的唯一完整防御，是确保凭据在任何时候都不作为明文存在于环境中。

VaultProof 使用分密钥架构将 API Key 分成加密份额。你的 CI/CD 管道从未拥有完整的密钥——只有份额。即使受损的工具读取环境的每个字节，它也找不到任何有用的东西。

🔑 密钥注册

你的 API Key 被分成 N 个份额。分发到单独的存储。每个份额单独使用是无用的。

🛡️ 运行时请求

你的应用请求 API 调用。VaultProof 代理收集份额，仅在内存中重建密钥几毫秒。

✓ 调用完成

API 调用成功。重建的密钥从内存中清零。明文密钥从未存在于你的应用环境中。

如果 Trivy 在此过程中运行，它将找不到任何东西。没有凭据可供窃取。当密钥不存在于运行时环境中时，攻击模型完全崩溃。
