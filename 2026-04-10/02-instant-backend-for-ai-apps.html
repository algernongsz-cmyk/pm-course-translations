Demos

当你选择Instant时，你会获得三个好处：

1. 你可以创建无限的应用，它们永远不会被冻结。
2. 你获得一个同步引擎，所以你的应用可以离线工作，是实时的，并且感觉很快。

无限应用

传统上，当你想要在线托管应用时，你要么为VM付费，要么受到限制。许多服务限制你可以创建的免费应用数量，并在它们空闲时冻结它们。解冻通常需要超过30秒，有时甚至几分钟。

我们认为这很糟糕。所以使用Instant，你可以创建任意数量的项目，我们永远不会冻结它们。

我们可以这样做，因为Instant被设计为多租户。当你创建新项目时，我们不会启动VM。我们只是在多租户实例中插入几行数据库记录。

如果你的应用不活跃，根本没有计算或内存成本。当它活跃时，开销只是几KB的额外RAM——相比之下VM需要数百MB。

这意味着你真正可以创建无限应用。事实上，这个过程如此高效，以至于我们可以在这篇文章中直接为你创建一个应用。无需注册。

如果你点击按钮，你会获得一个隔离的后端。

创建后端后，整个过程包括往返你的计算机只需要几百毫秒。

你会获得一个公共的App ID来标识你的后端，以及一个私有的Admin Token让你进行特权更改。这为你提供了关系数据库、同步引擎以及我们提到的额外服务，如auth和storage。

将无限应用与Agent结合，你将开始以不同的方式构建。今天你已经可以使用Agent创建大量应用。使用Instant，你永远不会被阻止将它们推向生产。

同步引擎

但一旦你创建了应用，如何让它变得更好？

构建传统的CRUD应用很容易。只需要让Agent连接一些数据库迁移、后端端点和客户端存储。但让这些应用变得令人愉悦很难。

将传统的CRUD应用与现代应用如Linear、Notion和Figma进行比较。现代应用是多人的，它们可以离线工作，而且感觉很快。如果你在Linear中更改一个todo，它会在任何地方更改。如果你在Notion中离线，你仍然可以标记你的文档。当你在Figma中为形状着色时，它不需要等待服务器，你只是看到它。

这类应用需要自定义基础设施。对于实时功能，你需要添加有状态的websocket服务器。对于离线模式，你需要在IndexedDB中存储缓存。对于乐观更新，你需要弄清楚如何在客户端应用和撤销变更。

Linear、Notion和Figma都构建了自定义基础设施来处理这个问题。作为一个行业，我们将他们的基础设施称为同步引擎[^2]。开发者编写UI并查询他们的数据，就好像它是本地可用的一样。同步引擎在后台处理所有数据管理。

如果现代应用需要同步引擎，那么你不应该每次都从头构建它们。

所以我们在Instant中构建了一个通用的同步引擎。每个应用默认都具有多人、离线模式和乐观更新。

你可以自己尝试。既然我们已经创建了隔离后端，让我们继续使用它：

你看到的是两个渲染todo应用的iframe。它们由你刚刚创建的后端驱动（我们将你的App ID传递给了iframe）。

现在如果你在一个iframe中添加todo，它将出现在另一个中。如果你离线，你可以进行更改，它们将同步在一起。你可以尝试降低网络质量，更改仍然感觉很快。

这就是todo应用后端代码的样子：

import { init, id } from '@instantdb/react';


const db = init({ appId: 'YOUR_APP_ID' });



function App() {

  const { data } = db.useQuery({ todos: {} });



  const addTodo = (text) =>

    db.transact(

      db.tx.todos[id()].update({ text, done: false }),

    );



  const toggleTodo = (todo) =>

    db.transact(

      db.tx.todos[todo.id].update({ done: !todo.done }),

    );



  return (

    
      todos={data?.todos ?? []}

      onAdd={addTodo}

      onToggle={toggleTodo}

    />

  );

}

大约25行。这甚至比你构建传统CRUD应用更简洁。你需要编写后端端点和前端存储。相反，你只需要在前端直接进行查询和事务。

db.useQuery让你编写关系查询，它们保持同步。db.transact让你进行更改，它可以离线工作。

这对作为构建者的你更好：代码可理解且易于维护。对你的用户更好：他们获得令人愉悦的应用。对你的Agent也更好。同步引擎是一个紧密的抽象[^3]，所以Agent可以使用它们编写更简洁的代码，使用更少的token，犯更少的错误。

额外服务

你看到了数据同步，但不止于此。应用通常需要比数据同步更多的功能。

例如，现在每个打开我们demo应用的人都看到同一组todo。如果我们想添加auth或权限怎么办？我们可能还想支持文件上传，或一个"谁在线"部分。或者也许我们添加一个AI助手，需要基础设施将token流式传输到客户端。

这些是大多数应用需要的常见功能。但通常我们需要组合不同的服务来获得它们。这不仅令人烦恼，还引入了新的复杂性级别。当你管理多个服务时，你管理多个数据源。

所以为了让增强你的应用更容易，我们在Instant内部内置了一堆常见服务。每个服务都被构建为一个单一集成系统的一部分工作。

为了了解这些服务，让我们再次看看我们的todo应用，但这次我们将添加文件上传支持：

传统做法是什么？我们首先在事务数据库中创建一个files表，并将其链接到todos。但我们需要存储实际的文件blob，所以可能会添加S3。

一旦添加了S3，我们需要处理多个数据源。例如，如果我们删除一个todo，我们需要运行后台工作器来删除S3中相应的blob。

使用Instant，所有这些都不是问题。

你默认获得文件存储，文件对象只是数据库中的行。它们就像任何其他实体：你可以创建它们，将它们链接到其他数据，并对它们运行实时查询。

这意味着你甚至可以创建CASCADE删除规则，所以你可以说"当你删除todos时，删除文件"。不需要后台工作器。不是多个数据源，你获得一个集成的数据库。共享基础设施在后台处理所有边缘情况[^4]。

这只是Instant Storage。你还获得Auth。你可以开箱即用地使用Magic Codes、OAuth和Guest Auth。而且当你的用户注册时，他们也只是数据库中的行。

如果你想共享光标、打字指示器或"谁在线"标记，你可以使用Instant Presence。

如果你需要共享持久流，你获得，嗯，Instant Streams。

如果你好奇，我们在recipes页面有一堆真实示例你可以玩。你会注意到大多数这些服务需要很少的设置和很少的代码。你和你的Agent都可以更快地行动，让你的应用功能丰富。你不必搜寻不同的提供商并处理双向数据同步。

架构

Instant的运作有三个独特之处。我们有Client SDK、Clojure后端和多租户数据库。

你的应用直接向Client SDK发送查询和事务。它负责离线解析你的查询，并在你进行事务时立即应用它们。

Client SDK然后与Clojure后端通信。Clojure后端保持查询实时。它接收事务并确定哪些客户端需要知道它们。它还实现所有额外服务：权限、auth、presence、storage和streams。

最后，Clojure后端向单个Postgres实例发送查询和事务。我们将Postgres视为多租户Triple store，并通过App ID逻辑上分离每个数据库。

这就是我们系统的草图。现在让我们深入了解。

Client SDK

Client SDK背后的设计由两个约束驱动：我们需要一个可以离线工作的系统，我们需要它支持乐观更新。

这是我们要达到的：

IndexedDB

让我们从最明显的框开始。如果我们想离线显示应用，我们需要一个地方让数据在刷新之间存活。

对于Web，你没有太多选择。IndexedDB是最佳候选。你可以存储许多MB的数据，你甚至有一些有限的查询能力。

所以我们选择了IndexedDB[^5]。下一个问题是，我们会在那里存储什么样的数据？

Triple store

考虑一个查询如"向我显示所有打开的todo及其附件"。这是你在Instant中编写它的方式：

{
  todos: {
    $: { where: { done: false } },
    attachments: { },
  },
}

如果我们只想要一个只读缓存，我们可以存储服务器返回给我们的任何东西。但我们不只想要一个只读缓存。

我们需要客户端在服务器确认之前响应操作。例如，如果用户添加一个新todo，我们的查询应该立即更新。

这意味着客户端需要理解查询。那么我们的客户端真正需要的是一个数据库本身。一个可以处理where子句（即'done is false'）和关系（'todos 和它们的附件'）的数据库。

一个选择是使用SQLite。我们可以在那里存储规范化的表——如todos和files——并对它们运行SQL。但这太重了。SQLite gzipped后大约300KB。对于大多数应用，添加如此重的依赖没有意义。

经过一番搜寻，我们发现了Triple store和Datalog。

Triple store让你将数据存储为[entity, attribute, value]元组。这是todo在Triple store中的样子：

这个统一结构可以建模属性和关系。一旦数据以这种方式存储，你可以使用Datalog对它进行查询。

Datalog是一个基于逻辑的查询引擎。这是它的样子：

语法看起来很奇怪，但Datalog很强大。它可以像SQL一样支持where子句和关系。而且它实现简单。事实上，你可以在不到一百行代码中编写一个基本的Datalog引擎[^6]。

所以我们构建了一个Triple store和一个Datalog引擎。这让我们可以完全在客户端中评估查询，而不必等待服务器。

如果用户创建一个新todo，我们有了立即重新运行查询并观察更改所需的一切。好吧，几乎。我们需要一种方法来对查询应用更改。

Pending Queue

我们不能只是就地修改结果。我们也需要考虑服务器。

例如，如果服务器拒绝我们的事务会发生什么？如果我们修改了查询结果，就没有办法撤销更改。[^7]

这就是Pending Queue的用武之地。当用户进行更改时，我们不直接将其应用到Triple store。相反，我们在单独的队列中跟踪更改。

为了满足任何查询，我们可以将pending更改应用到我们的triple store，并查看结果：

这个选择推动我们使Triple store不可变。这样我们可以应用更改并产生一个新的Triple store，而不是修改已提交的那个。为了使这工作，我们用mutative包装transact API，这是一个用于JavaScript不可变更改的库[^8]。

有了这个我们就有了撤销。如果服务器返回失败，我们只需从pending队列中删除更改，撤销就开箱即用。

额外：InstaQL

你可能注意到Instant查询看起来不像Datalog。相反它们是用我们称为InstaQL的语言编写的：

{
  todos: {
    $: { where: { done: false } },
    attachments: { },
  },
}

我们这样做是因为我们认为应用查询数据最符合人体工程学的方式是描述它们正在寻找的响应形状。

这个想法深受GraphQL启发。与我们实现的主要区别是语法糖。InstaQL不是引入特定的语法，而是建立在普通JavaScript对象之上。这个选择让用户跳过构建步骤，并让他们可以编程地生成查询[^9]。

Reactor

有了这个，我们对Client SDK有了某种程度上完整的了解！

用户编写InstaQL查询，它被转换为Datalog。这些查询由Triple store满足，它结合了来自pending队列的更改。数据被缓存到IndexedDB。

仅从两个约束就产生了许多有趣的选择！

客户端上的最后一个问题是：所有这些框如何结合在一起？

这就是Reactor的用武之地。它是协调所有这些不同进程的主状态机。当应用想要查询时，Reactor负责查看IndexedDB并与服务器通信。它处理互联网离线或pending更改失败的情况。

Reactor通过websocket与服务器通信。它发送查询和事务请求，服务器发送来自数据库的结果和新数据。

这将我们带到服务器。

Clojure后端

后端背后的设计由两个约束驱动：我们需要使查询具有反应性，我们需要公平对待多租户资源。

这是系统大致的样子：

Query Store

让我们从思考用户请求查询时会发生什么开始。

首先服务器可以继续并询问数据库。在无状态系统中，那将是故事的大致终点。我们可以返回我们的响应并收工。

但记住，我们的查询必须是反应性的。为此我们需要一个地方来存储哪些用户进行了哪些查询。这就是Query Store的用途：

如果我们只跟踪查询和请求它们的socket连接，原则上我们就有了使应用具有反应性所需的一切。例如我们可以tail每个事务并刷新每个查询。这会工作，但我们的数据库会被大量垃圾请求轰炸。

理想情况下，我们应该只更改需要更改的查询。

Grouped Queues

由于我们的数据库是多租户的，我们的WAL包含来自多个应用的更新。

为了使失效算法工作，单个应用内的事务必须按顺序串行处理。但是，我们当然可以跨不同应用并行化失效。

我们需要某种方法来保证单个应用内的顺序并跨应用并行化。我们还需要确保一个高流量应用不会独占所有资源。

这就是Grouped Queue抽象的用武之地：

每个应用获得自己的子队列。这保证了特定应用的所有项目被串行处理。

然而，Worker可以从多个不同的子队列中获取。这让我们可以跨应用并行化失效。

当我们向grouped queue推送一个WAL条目时，它被添加到应用的子队列，但子队列的全局顺序不会改变。这使得即使一个应用每秒添加数千个项目，其他应用仍然有同等的机会被invalidator选中。

这个数据结构对我们来说变得非常有用，并且渗入了整个代码库，包括Session Manager。

Session Manager，以及对Clojure和JVM的赞美

这带我们到系统内的主要协调器。当Client SDK打开一个websocket连接时，是session manager接收消息：

Session Manager的工作是将所有东西粘合在一起。它进行反应性查询，运行权限，并将请求传递给其他服务。

注意Grouped Queue抽象也在这里出现。如果不同的客户端开始轰炸后端，Grouped Queue确保尽可能多地并行化，并防止一个坏的socket独占所有资源。

有了这个，可能是暂停并赞美Clojure和JVM的合适地方。它们对我们构建这个基础设施是一个巨大的胜利。

首先，Clojure有很棒的并发原语并有真正的线程。这让我们可以用更大的机器进一步扩展，并帮助我们避免过早拆分系统。抽象也非常简单且易于组合。例如我们的grouped queue只有215行代码[^13]

其次，JVM有一个繁荣的生态系统，我们真的喜欢这些库。例如，我们需要一种方法让用户在Instant内定义权限。我们想要一种快速且易于沙箱化的语言。经过一些搜索，我们发现了Google的CEL。幸运的是CEL Java可用，我们可以直接拿来使用。

第三，Clojure非常适合DSL和实验性编程。当我们开始构建Instant时，我们需要发现许多这些抽象，在REPL中玩它们是至关重要的。

许多人嘲笑DSL，但我认为没有它们我们无法构建Instant。一个例子：多租户查询。我们需要使我们的数据库多租户。为此我们需要编写一些相当复杂的SQL。我们不是手动做这件事，而是制作了一个DSL，既使推理变得容易，又保证你可以传入App ID。

这将我们带到多租户数据库。

多租户数据库

我们的数据库也由两个约束驱动：我们需要一种廉价启动新数据库的方法，我们需要它是关系型的。

这是我们达到的：

Triples表

让我们从问题开始：我们如何让用户创建许多不同的数据库？

最直接的路径是启动Postgres VM。但如我们所提到的，VM在RAM中带来大量开销。如果你在启动VM，没有可持续的方式来支持无限应用。

另一个选择是使用Postgres schema。我们可以为不同的应用创建不同的表，然后保留谁可以看到什么的映射。这会工作，但Postgres没有被设计为很好地随表扩展。从我们的研究中我们看到大约6000个表后，Postgres开始有问题：你会遇到磁盘上创建多少文件的问题，pg_dump和autovacuum开始失败。

这有道理。平均Postgres应用有几个大表，而不是许多小表，这意味着大表被优化。好吧，如果大表工作，如果我们将这个问题重新框架为一个巨大的表呢？

这将我们带回...Triple store！

它们在客户端工作得很好，因为它们是一个支持关系查询的简单DB。我们认为这在Postgres中对我们也可能工作得很好。所以我们添加了一个triples表：

所有数据存在于单个triples表中，它们通过app_id逻辑上隔离。

例如，如果我们想从应用blog获取post_1，我们可以生成一个大致如下的SQL查询：

select *
from triples
where app_id = 'blog' and entity_id = 'post_1' and attr_id in (posts/id, posts/title)

有了这个，创建新数据库实际上是免费的。正如我们在demo中提到的，这只是数据库中的几行。

#### 意外的收益

我们的选择也带来了一些意外的收益。

由于我们自己管理列，我们能够优化开发者体验。

例如，Postgres在你创建列时锁定表。由于我们自己实现列，我们可以使它们无锁。

当你在Postgres中删除一列时，数据就没了。但我们认为在Agent世界中这太危险了。所以我们在列级别实现了软删除。即使一个流氓Agent删除了你的列，你也可以撤销它并在几毫秒内取回所有数据。

这些是收益，但当然也有成本。

部分索引

考虑一个说"我想让我的posts有一个独特的'slug'"的用户。在Postgres中创建唯一列很容易。但由于我们正在实现自己的列，我们必须自己做到这一点。

这就是部分索引的救赎。我们可以向我们的triples表添加布尔标记：

table_name: triples
app_id | entity_id | attr_id | value | column_unique | ...

一旦我们有了那个，我们可以为整个表创建一个部分索引，通过标记翻转启用：

create unique index unique_columns
  on triples(app_id, column, value) where column_unique

现在如果用户尝试插入两个具有相同slug的posts：

app_id  | object_id | column | value   | column_unique
'blog' | 1         | 'slug' | 'hello' | true
'blog' | 2         | 'slug' | 'hello' | true
unique_columns索引触发并阻止它！

这同样的技巧使我们的查询更高效。例如，如果我们想找到slug为'hello'的posts，我们可以生成这个查询：

select entity_id
from triples
where app_id = 'blog' and attr_id = 'slug' and value = 'hello' and column_unique;

我们可以将这个模式扩展到一系列查询：唯一列、索引、日期、引用等。

仅使用部分索引并依赖Postgres进行正确的查询对我们来说工作了一段时间。但在我们达到几亿个元组的规模后，Postgres开始有麻烦。

Count-Min Sketches

如果你是正在阅读这篇的Postgres专家，你可能看到那个triples表时停顿了一下。在Postgres圈子中，这被称为EAV模式，通常不被鼓励。

它不被鼓励是因为Postgres依赖表和列来获取统计信息。

这些统计信息是让查询计划器决定哪些索引最高效以及以什么顺序做哪些连接的原因。

一旦你将所有数据保留在一个表中，Postgres就会失去关于数据集中底层频率的信息。它无法区分有10个不同值的列和有1000万个不同值的列。

为了解决这个问题，我们开始跟踪我们的统计信息。我们使用一种称为count-min sketches的数据结构，它帮助我们估计列的频率。如果你好奇它是如何工作的，我们写了一篇关于它的文章[^14]。

我们可以将这些统计信息提供给我们的查询引擎，并再次使这些查询高效。
