## 无限应用

传统上，当你想要在线托管应用时，你要么为 VM 付费，要么受到限制。许多服务限制你可以创建的免费应用数量，并在它们空闲时冻结它们。解冻通常需要超过 30 秒，有时甚至需要几分钟。

我们认为这很糟糕。所以使用 Instant，你可以创建任意数量的项目，我们永远不会冻结它们。

我们可以做到这一点，因为 Instant 被设计为多租户的。当你创建一个新项目时，我们不会启动 VM。我们只是在多租户实例中插入几行数据库。

如果你的应用处于非活动状态，根本没有任何计算或内存成本。当它处于活动状态时，开销只有几 KB 的额外 RAM——而不是 VM 所需的数百 MB。

这意味着你可以真正创建无限的应用。事实上，这个过程非常高效，我们可以在这篇文章中为你创建一个应用。无需注册。

## 同步引擎

但是一旦你创建了一个应用，如何让它变得优秀？

构建传统的 CRUD 应用很容易。只需让 Agent 连接一些数据库迁移、后端端点和客户端存储。但是让这些应用变得*令人愉悦*很难。

将传统的 CRUD 应用与 Linear、Notion 和 Figma 等现代应用进行比较。现代应用是多玩家的，它们可以离线工作，而且感觉很快。如果你在 Linear 中更改一个 todo，它会在任何地方更改。如果你在 Notion 中离线，你仍然可以标记你的文档。当你在 Figma 中为形状着色时，它不会等待服务器，你只是看到它。

这些类型的应用需要自定义基础设施。对于实时功能，你添加有状态的 websocket 服务器。对于离线模式，你在 IndexedDB 中存储缓存。对于乐观更新，你弄清楚如何在客户端应用和撤消变更。

Linear、Notion 和 Figma 都构建了自定义基础设施来处理这个问题。作为一个行业，我们将它们的基础设施称为同步引擎。开发者编写 UI 并查询他们的数据，就像它本地可用一样。同步引擎在后台处理所有数据管理。

如果现代应用需要同步引擎，那么你不应该每次都从头构建它们。

所以我们在 Instant 中构建了一个通用同步引擎。每个应用默认都具有多玩家、离线模式和乐观更新功能。

## 其他服务

你看到了数据同步，但这并没有结束。应用通常需要的不仅仅是数据同步。

例如，现在每个打开我们 demo 应用的人都会看到同一组 todo。如果我们想添加 auth 或权限怎么办？我们可能还想支持文件上传，或"谁在线"部分。或者也许我们添加一个 AI 助手，并需要基础设施将 token 流式传输到客户端。

这些是大多数应用需要的常见功能。但我们经常需要将不同的服务串联起来才能获得它们。这不仅令人讨厌，而且引入了新的复杂性级别。当你管理多个服务时，你管理多个真实来源。

所以为了让你的应用更容易增强，我们在 Instant 内部内置了一堆常见服务。每个服务都构建为作为一个单一的集成系统协同工作。

为了了解这些服务，让我们再次看看我们的 todo 应用，但这次我们将添加对文件上传的支持：

传统上怎么做？我们首先在事务数据库中创建一个 `files` 表，并将其链接到 `todos`。但我们需要存储实际的文件 blob，所以我们可能会添加 S3。

一旦我们添加 S3，我们就有多个真实来源需要处理。例如，如果我们删除一个 todo，我们需要运行一个后台工作器来删除 S3 中相应的 blob。

使用 Instant，所有这些都不是问题。

你默认获得 File Storage，文件对象只是数据库中的行。它们就像任何其他实体一样：你可以创建它们，将它们链接到其他数据，并对它们运行实时查询。

这意味着你甚至可以创建 CASCADE 删除规则，所以你可以说"当你删除 todo 时，删除文件"。不需要后台工作器。而不是多个真实来源，你获得一个集成的数据库。共享基础设施在后台处理所有边缘情况。

这只是 Instant Storage。你还可以获得 Auth。你可以开箱即用地使用 Magic Codes、OAuth 和 Guest Auth。另外，当你的用户注册时，他们也只是数据库中的行。

如果你想共享光标、输入指示器或"谁在线"标记，你可以使用 Instant Presence。

如果你需要共享持久流，你获得，嗯，Instant Streams。

如果你好奇，我们在 [recipes](/recipes) 页面有一堆真实示例，你可以尝试。你会注意到这些服务大多数需要很少的设置和很少的代码。你和你的 Agent 都可以更快地移动并使你的应用功能丰富。你不必搜寻不同的提供商并处理双向数据同步。

## 架构

Instant 的运作方式有三个独特之处。我们有 Client SDK、Clojure Backend 和 Multi-Tenant Database。

你的应用直接向 Client SDK 发送查询和事务。它负责离线解析你的查询，并在你进行更改后立即应用事务。

Client SDK 然后与 Clojure Backend 通信。Clojure Backend 保持查询实时。它接收事务并确定哪些客户端需要知道它们。它还实现所有其他服务：权限、auth、presence、storage 和 streams。

最后，Clojure Backend 向单个 Postgres Instance 发送查询和事务。我们将 Postgres 视为多租户 Triple store，并按 App ID 逻辑分离每个数据库。

这就是我们系统的草图。现在让我们更深入。

Client SDK 背后的设计受两个约束驱动：我们需要一个可以离线工作的系统，我们需要它支持乐观更新。

## IndexedDB

让我们从最明显的框开始。如果我们想离线显示应用，我们需要一个地方让数据在刷新后仍然存在。

对于 Web，你没有太多选择。IndexedDB 是最佳候选者。你可以存储许多 MB 的数据，你甚至有一些有限的查询能力。

所以我们选择了 IndexedDB。下一个问题是，我们会在那里存储什么样的数据？

## Triple store

考虑一个像"显示所有打开的 todo 及其附件"的查询。这是你在 Instant 中编写它的方式：

```typescript
{
  todos: {
    $: { where: { done: false } },
    attachments: { },
  },
}
```

如果我们只想要一个只读缓存，我们可以存储服务器返回给我们的任何内容。但我们不仅仅想要一个只读缓存。

我们需要客户端在服务器确认之前响应操作。例如，如果用户添加一个新的 todo，我们的查询应该立即更新。

这意味着客户端需要理解查询。那么我们的客户端真正需要的是一个数据库本身。一个可以处理 where 子句（即，'done is false'）和关系（'todo 及其附件'）的数据库。

一个选择是使用 SQLite。我们可以在那里存储规范化表——比如 `todos` 和 `files`——并在它们上运行 SQL。但这太重了。SQLite 大约是 300 KB gzipped。对于大多数应用来说，添加如此重的依赖项是没有意义的。

如果用户创建一个新的 todo，我们有了重新运行查询并立即观察更改所需的一切。嗯，几乎。我们需要一种方法来对查询应用更改。

## Pending Queue

我们不能只是就地更改结果。我们也必须考虑服务器。

例如，如果服务器拒绝我们的事务会发生什么？如果我们更改了查询结果，就没有办法让我们撤消更改。

这就是 Pending Queue 的用武之地。当用户进行更改时，我们不会直接将其应用于 Triple store。相反，我们在一个单独的队列中跟踪更改。

这个选择推动我们使 Triple store 不可变。这样我们可以应用更改并生成一个新的 Triple store，而不是更改已提交的那个。为了使这工作，我们使用 mutative 包装 transact API，这是一个用于 Javascript 中不可变更改的库。

有了这个，我们有了撤消功能。如果服务器返回失败，我们只需从待处理队列中删除更改，撤消就开箱即用了。

## 额外内容：InstaQL

你可能已经注意到 Instant 查询看起来不像 Datalog。相反，它们用我们称为 InstaQL 的语言编写：

```typescript
{
  todos: {
    $: { where: { done: false } },
    attachments: { },
  },
}
```

我们这样做是因为我们认为应用查询数据最符合人体工程学的方式是描述他们正在寻找的响应的形状。

这个想法受到 GraphQL 的很大启发。我们实现的主要区别是语法糖。不是引入特定的语法，InstaQL 构建在普通 javascript 对象之上。这个选择让用户跳过构建步骤，并让他们以编程方式生成查询。

## Reactor

有了这个，我们对 Client SDK 有了一个相当完整的了解！

用户编写 InstaQL 查询，它们被转换为 Datalog。这些查询由 Triple stores 满足，它们组合来自待处理队列的更改。数据被缓存到 IndexedDB。

从仅仅两个约束生成了很多有趣的选择！

客户端上的最后一个问题是：所有这些框如何联系在一起？

这就是 Reactor 的用武之地。它是协调所有这些不同进程的主要状态机。当应用想要查询时，Reactor 负责查看 IndexedDB，并与服务器通信。它处理互联网离线或待处理更改失败的情况。

Reactor 通过 websocket 与服务器通信。它发送查询和事务请求，服务器发送结果和数据库的新颖性。

这把我们带到了服务器。

## Grouped Queues

由于我们的数据库是多租户的，我们的 WAL 包含来自多个应用的更新。

为了使失效算法工作，单个应用*内*的事务必须按顺序串行处理。但是，我们当然可以在*不同*应用之间并行化失效。

我们需要某种方法来保证单个应用内的顺序并在应用之间并行化。我们还需要确保一个高流量应用不会占用所有资源。

这就是 Grouped Queue 抽象的用武之地：

每个应用获得自己的子队列。这保证了特定应用的所有项目都被串行处理。

然而，工作者可以从多个不同的子队列获取。这让我们可以跨应用并行化失效。

当我们把 WAL 条目推入分组队列时，它被添加到应用的子队列，但子队列的全局顺序不会改变。这使得即使一个应用每秒添加数千个项目，其他应用仍然有平等的机会被 invalidator 选中。

这个数据结构对我们来说非常有用，并且渗透到了整个代码库，包括 Session Manager。

## Session Manager，以及对 Clojure 和 JVM 的赞扬

这把我们带到了系统内部的主要协调器。当 Client SDK 打开 websocket 连接时，session manager 接收消息：

Session Manager 的工作是将一切联系在一起。它进行响应式查询，运行权限，并将请求传递给其他服务。

注意 Grouped Queue 抽象也出现在这里。如果不同的客户端开始轰炸后端，Grouped Queue 确保尽可能多地并行化，并防止一个坏的 socket 占用所有资源。

有了这个，这可能是暂停并赞扬 Clojure 和 JVM 的正确地方。它们对我们构建这个基础设施来说是一个巨大的胜利。

首先，Clojure 带有很棒的并发原语并且有真正的线程。这让我们可以用更大的机器进一步扩展，并帮助我们避免过早拆分系统。抽象也非常简单且易于组合。例如，我们的分组队列只有 215 行代码。

其次，JVM 有一个繁荣的生态系统，我们真的很喜欢这些库。例如，我们需要一种方法让用户在 Instant 中定义权限。我们想要一种快速且易于沙箱化的语言。经过一番搜索，我们发现了 Google 的 CEL。幸运的是 CEL Java 可用，我们可以直接拿来使用。

第三，Clojure 很适合 DSL 和实验性编程。当我们开始构建 Instant 时，我们需要发现很多这些抽象，在 REPL 中玩弄它们是至关重要的。

许多人嘲笑 DSL，但我认为没有它们我们就无法构建 Instant。例子：多租户查询。我们需要使我们的数据库多租户。为此我们需要编写一些相当复杂的 SQL。我们不是手工完成这个，而是制作了一个 DSL，既容易推理，又保证你可以传入 App ID。

这把我们带到了 Multi-Tenant Database。

## Triples Table

让我们从这个问题开始：我们如何让用户创建许多不同的数据库？

最直接的路径是启动 Postgres VM。但正如我们提到的，VM 带有大量的 RAM 开销。如果你启动 VM，就没有可持续的方法支持无限应用。

另一个选择是使用 Postgres schemas。我们可以为不同的应用创建不同的表，然后保留谁可以看到什么的映射。这会工作，但 Postgres 不是设计为与表一起良好扩展的。从我们的研究中，我们看到在大约 6000 个表之后，Postgres 开始出现问题：你会在磁盘上创建多少文件方面遇到问题，pg_dump 和 autovacuum 开始失败。

这是有道理的。平均 Postgres 应用有几个大表，而不是许多小表，这意味着大表得到优化。嗯，如果大表工作，如果我们把这个问题重新框架成一个巨大的表呢？

这把我们带回了...Triple stores！

它们在客户端上工作得很好，因为它们是支持关系查询的简单 DB。我们认为这在 Postgres 中也可能对我们工作得很好。所以我们添加了一个 `triples` 表：

所有数据都存在于单个 `triples` 表中，它们按 `app_id` 逻辑隔离。

例如，如果我们想从应用 `blog` 获取 `post_1`，我们可以生成一个大致如下所示的 SQL 查询：

```sql
select *
from triples
where app_id = 'blog' and entity_id = 'post_1' and attr_id in (posts/id, posts/title)
```

有了这个，创建新数据库实际上是免费的。正如我们在 demo 中提到的，这只是数据库中的几行。

### 意外的好处

我们的选择也带来了一些意外的好处。

由于我们自己管理列，我们能够优化开发者体验。

例如，Postgres 在创建列时锁定表。由于我们自己实现列，我们可以使它们无锁。

当你在 Postgres 中删除列时，数据就没了。但我们认为在 Agent 的世界中这太危险了。所以我们在列级别实现了软删除。即使一个流氓 Agent 删除了你的列，你也可以撤消它并在几毫秒内恢复所有数据。

这些是好处，但当然也有成本。

## Partial Indexes

考虑一个用户说，"我想让我的帖子有一个唯一的'slug'"。在 Postgres 中很容易创建唯一列。但由于我们正在实现自己的列，我们必须自己做。

这就是 partial indexes 救援的地方。我们可以向我们的 `triples` 表添加布尔标记：

```
table_name: triples
app_id | entity_id | attr_id | value | column_unique | ...
```

一旦我们有了那个，我们可以为整个表创建一个 partial index，由标记翻转：

```sql
create unique index unique_columns
  on triples(app_id, column, value) where column_unique
```

现在如果用户尝试插入两个具有相同 slug 的帖子：

```sql
app_id  | object_id | column | value   | column_unique
'blog' | 1         | 'slug' | 'hello' | true
'blog' | 2         | 'slug' | 'hello' | true
```

`unique_columns` index 触发并阻止它！

这个相同的技巧使我们的查询更高效。例如，如果我们想找到具有 slug 'hello' 的帖子，我们可以生成这个查询：

```sql
select entity_id
from triples
where app_id = 'blog' and attr_id = 'slug' and value = 'hello' and column_unique;
```

我们可以将这个模式扩展到各种查询：唯一列、索引、日期、引用等。

只使用 partial indexes 并依赖 Postgres 进行正确的查询对我们来说工作得很好一段时间。但在我们达到几亿个元组的规模后，Postgres 开始遇到麻烦。

## Count-Min Sketches

如果你是一个阅读这篇文章的 Postgres 专家，你可能已经在看那个 triples 表时停顿了一下。在 Postgres 圈子中，这被称为 EAV 模式，通常是不鼓励的。

它不鼓励是因为 Postgres 依赖表和列来获取统计信息。

这些统计信息是让查询规划器决定哪些索引最有效以及以什么顺序进行连接的原因。

一旦你将所有数据保存在一个表中，Postgres 就会失去关于数据集中底层频率的信息。它无法区分具有 10 个不同值的列和具有 1000 万个不同值的列。

为了解决这个问题，我们开始跟踪我们的统计信息。我们使用一种称为 count-min sketches 的数据结构，它帮助我们估计列的频率。如果你对这如何工作感到好奇，我们写了一篇关于它的文章。

我们可以将这些统计信息提供给我们的查询引擎，并使这些查询再次高效。
